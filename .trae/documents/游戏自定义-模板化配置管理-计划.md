## Summary

将现有 `/config`“配置管理”改造为“游戏自定义”，在同一入口下支持：
1) 管理/新建规则、棋盘、卡牌（保留现有版本化能力）；2) 新增“模板”（规则+棋盘+卡牌 的发布版本组合）并进行版本化（草稿/发布/回滚）；3) 模板贯穿“创建房间 + 房间 lobby 配置”两处使用，其中“创建房间”仅允许选择模板（不再分别选择规则/棋盘/卡牌）。

  - 配置服务（list/create/updateDraft/publish/rollback/resolvePublished*）：[config-service.ts](file:///e:/code/ProjectNeoBlock/apps/web/lib/config-service.ts)
- 配置管理 UI
  - `/config` 列表页：分三组展示 rules/board/cards + 新建配置文档；含“返回首页”按钮：[config/page.tsx](file:///e:/code/ProjectNeoBlock/apps/web/app/config/page.tsx)
  - `/config/[docId]` 详情页：rules 用表单，board/cards 用 JSON 文本；保存草稿/发布/回滚：[config/[docId]/page.tsx](file:///e:/code/ProjectNeoBlock/apps/web/app/config/%5BdocId%5D/page.tsx)
- 已发布版本列表接口：仅 rules/boards/cards：[api/config/published](file:///e:/code/ProjectNeoBlock/apps/web/app/api/config/published/route.ts)

- `/api/room/create` 与 `/api/room/config` 只接受三项版本号（rulesetVersionId/boardVersionId/cardsVersionId），并校验它们必须存在且为“已发布版本”：[api/room/create](file:///e:/code/ProjectNeoBlock/apps/web/app/api/room/create/route.ts)、[api/room/config](file:///e:/code/ProjectNeoBlock/apps/web/app/api/room/config/route.ts)

## Decisions (Locked)

来自用户确认：
- 模板引用三类配置时，绑定“发布版本ID”（published versionId），以保证可复现与冻结组合。
- 模板需要“全链路”使用：
  - 创建房间时“只能选择模板”（不再分别选择规则/棋盘/卡牌）
  - 房间 lobby 也支持选择模板一键设置三项版本
- `/config` 页面结构改为内部 Tabs：
  - “模板”Tab：模板管理
  - “配置”Tab：规则/棋盘/卡牌 管理
- `/config` 页面移除“返回首页”按钮（依赖全局 Tabs 导航）

## Proposed Changes

### 1) 新增模板配置类型（数据模型 + 校验）

目标：将模板当作第四类 ConfigDoc(kind=template)，同样支持 draft/published/rollback。

改动点：
- 扩展配置 kind：
  - `ConfigKind` 从 `'rules'|'board'|'cards'` 扩展为 `'rules'|'board'|'cards'|'template'`
  - 新增 `TemplateConfig` 数据结构：
    - `rulesVersionId: string`
    - `boardVersionId: string`
    - `cardsVersionId: string`
- 新增 `validateTemplateConfig`（仅校验字段类型与非空）
- 扩展 `validateConfigByKind` 对 template 分发
  - 文件：[config.ts](file:///e:/code/ProjectNeoBlock/apps/web/lib/config.ts)、[config-service.ts](file:///e:/code/ProjectNeoBlock/apps/web/lib/config-service.ts)

### 2) 模板的“引用合法性”校验（必须引用已发布版本）

由于模板数据引用的是“发布版本ID”，仅做结构校验不够；需要在写入草稿/发布时验证版本存在性与发布状态。

改动点（在 `lib/config-service.ts` 内实现，具备 AppData 上下文）：
- 新增 helper：`isPublishedVersionId(data, kind, versionId)` / `getDocByPublishedVersionId(...)`
- `updateDraft`：
  - 当 `doc.kind === 'template'` 时：
    - 先 `validateTemplateConfig` 得到 TemplateConfig
    - 再校验三项 versionId 在 `data.configDocs` 中确实存在且为对应 kind 的 `publishedVersionId`
- `publishDoc`：
  - 当 `doc.kind === 'template'` 时，同样做“引用合法性”校验，避免发布非法模板
- `ensureSeedConfigs`：
  - 在已有 builtin rules/board/cards（含 builtin:board-full）seed 后，新增 1 个 seed 模板（例如 `builtin:template-standard`）：
    - draftData 引用当前可用的 published rules + board-full + published cards
    - 并发布，使系统默认就有至少一个模板可用（满足“创建房间只能选模板”的前置条件）
  - 文件：[config-service.ts](file:///e:/code/ProjectNeoBlock/apps/web/lib/config-service.ts)

### 3) 存储与反序列化支持 template kind

- `AppData.configDocs` 仍复用现有结构，但 `coerceConfigDoc` 需要允许 `kind === 'template'`
  - 文件：[store.ts](file:///e:/code/ProjectNeoBlock/apps/web/lib/store.ts)

### 4) 配置 API 扩展：支持 template 的 list/get/published/doc/draft/publish/rollback

改动点：
- `/api/config/list`：
  - kind 参数扩展支持 template
  - 返回 docs 列表里 kind 允许 template
- `/api/config/doc`：
  - Body.kind 扩展支持 template
  - 创建 template doc 的初始草稿数据：默认引用“当前最新 published”的 rules/board/cards（优先 board-full）
- `/api/config/published`：
  - 返回值新增 `templates: { docId,name,versionId,updatedAtMs }[]`
  - 文件：
    - [api/config/list](file:///e:/code/ProjectNeoBlock/apps/web/app/api/config/list/route.ts)
    - [api/config/doc](file:///e:/code/ProjectNeoBlock/apps/web/app/api/config/doc/route.ts)
    - [api/config/published](file:///e:/code/ProjectNeoBlock/apps/web/app/api/config/published/route.ts)

### 5) “游戏自定义(/config)”列表页 UI：内部 Tabs（模板 / 配置）

目标：把现有“配置管理”列表页改成“游戏自定义”，并将模板与三类配置拆成两个 Tab。

改动点（`apps/web/app/config/page.tsx`）：
- 标题从“配置管理”改为“游戏自定义”
- 移除“返回首页”按钮，仅保留“刷新列表”
- 页面内增加两个 Tab（用按钮切换 state，无需新增路由）：
  - 模板 Tab：
    - 新建模板（名称 + 选择 rules/board/cards 的 published 版本下拉）
    - 模板文档列表（进入 `/config/{docId}`）
  - 配置 Tab：
    - 保留现有“新建配置文档”（rules/board/cards）
    - 保留现有三组列表展示
- 数据加载策略：
  - `GET /api/config/list` 拉所有 docs，按 kind 分组（rules/board/cards/template）
  - `GET /api/config/published` 用于模板创建时的三个下拉

### 6) `/config/[docId]` 详情页 UI：新增 template 编辑器

目标：模板文档详情页支持：
- 选择三项“发布版本ID”（下拉）
- 保存草稿 / 发布 / 回滚（复用现有按钮与 API）
- 预览区展示组合摘要（显示选择的 rules/board/cards 名称 + versionId）

改动点（`apps/web/app/config/[docId]/page.tsx`）：
- `ConfigDocResponse.doc.kind` 扩展 template
- `Preview` 组件扩展 template：
  - 对 template data 做 `validateTemplateConfig`，并展示：rulesVersionId/boardVersionId/cardsVersionId
  - 为了展示名称：进入页面时额外拉 `/api/config/published`，把 versionId -> name 建 map（仅展示用）
- 编辑区：
  - kind === template 时，不再用 JSON 文本
  - 用 3 个 `<select>` 分别选择 published rules/board/cards 的 versionId
  - 保存时提交 `{ rulesVersionId, boardVersionId, cardsVersionId }`

### 7) 创建房间：只能选择模板

目标：创建房间 API 与首页 UI 改为“模板驱动”：
- 首页“快速创建”改为“选择模板创建”（下拉模板 + 创建）
- 不再在首页客户端拼三项 versionId

改动点：
- 扩展 `RoomConfig`：
  - 新增可选字段 `templateVersionId?: string`（保存用户选的模板发布版本，便于房间页展示/二次编辑）
  - 文件：[store.ts](file:///e:/code/ProjectNeoBlock/apps/web/lib/store.ts)、[room.ts](file:///e:/code/ProjectNeoBlock/apps/web/lib/room.ts)
- `/api/room/create`：
  - 接受 `config.templateVersionId`
  - 若提供 templateVersionId：
    - 校验它确实是 template doc 的 publishedVersionId
    - 读取模板数据（rulesVersionId/boardVersionId/cardsVersionId）写入 room.config 的三项版本
    - 同时保存 `templateVersionId`
  - 若未提供 templateVersionId：为了兼容旧调用，可继续走原逻辑（但首页 UI 将不再这么调用）
  - 文件：[api/room/create](file:///e:/code/ProjectNeoBlock/apps/web/app/api/room/create/route.ts)
- 首页 UI（`apps/web/app/page.tsx`）：
  - `useEffect` 拉 `/api/config/published` 时同时取 `templates`
  - “快速创建”卡片：增加模板下拉；创建房间 body.config 只带 `templateVersionId`（以及 maxPlayers/turnTimeSec/enableAuto/enableAI）
  - “自定义”卡片：跳转 `/config`，默认打开模板 Tab（例如 `router.push('/config?tab=template')`；或以本地 state 默认模板 Tab）

### 8) 房间 lobby：一键选择模板设置三项版本

目标：房间页（当前实际使用的是 WS 版 `RoomPage`）在 lobby 且房主时提供“选择模板应用”能力，服务端落库后继续复用既有 WS 同步流程。

改动点：
- `/api/room/config`：
  - 支持 `config.templateVersionId`：
    - 解析模板数据并写入 room.config 三项版本 + templateVersionId
    - 仍然校验三项版本必须为 published
  - 文件：[api/room/config](file:///e:/code/ProjectNeoBlock/apps/web/app/api/room/config/route.ts)
- 房间页 UI（`apps/web/app/room/[code]/page.tsx`）：
  - lobby 且房主：展示模板下拉 + “应用模板”按钮
  - 点击后调用 `/api/room/config`，仅提交 `{ roomCode, config: { templateVersionId } }`
  - 显示当前 room.config.templateVersionId（若存在）对应的模板名称（用 `/api/config/published` 映射）

## Assumptions

- 模板只组合三类配置（rules/board/cards），不额外包含房间参数（maxPlayers/turnTimeSec/enableAuto/enableAI）。
- 模板的 draft/publish/rollback 语义与现有三类配置一致。
- 为满足“创建房间只能选模板”，会提供至少一个 seed 模板，避免空系统无法创建房间。

## Verification Steps

1) Typecheck：`npm -w @neoblock/web run typecheck`
2) API 合法性（手测）：
   - `/api/config/published` 返回包含 `templates`
   - 新建模板 doc → 保存草稿 → 发布 → 回滚流程可用
   - 模板引用非法 versionId 时，`/api/config/draft` 与 `/api/config/publish` 返回 VALIDATION_FAILED（或等价错误）
3) UI 流程（手测）：
   - `/config` 默认进入“模板”Tab；切换到“配置”Tab 仍可管理 rules/board/cards
   - 首页创建房间只能选择模板并成功创建
   - 房间 lobby 选择模板应用后，room.config 三项版本变更并能正常开局（沿用既有 ws 同步）
